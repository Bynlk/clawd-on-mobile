## 一、安全性评估 — 86/100 ✅ 良好

### 做得好的

- **加密存储**: `EncryptedSharedPreferences` (AES-256-GCM) 保护 token、连接配置
- **TOFU 证书锁定**: `CertificateVerifier` 提取 SHA-256 指纹，非 LAN 连接强制 TLS
- **NetworkSecurityConfig**: LAN 地址允许明文，其余强制 TLS，配置规范
- **广播安全**: `ApprovalReceiver` 静态注册于 AndroidManifest 并声明 `exported=false`（静态 receiver 不需要 `RECEIVER_NOT_EXPORTED` 标志）；`FloatingPetService` 内的动态 receiver 使用了 `RECEIVER_NOT_EXPORTED`
- **Token 不入 URL**: 认证通过 `Authorization: Bearer` header，URL 中不含 token
- **主机验证**: `ConnectionConfig.isValidHost` 只允许 IP / localhost / mDNS，拒绝任意域名
- **WorkManager 审批**: 通知栏 approve/deny 通过 WorkManager + 指数退避，避免 ANR

### 风险点

| 风险 | 严重度 | 说明 |
|------|--------|------|
| WebView JS 启用 | 中 | `FloatingPetView` 开启 `javaScriptEnabled`，虽然只加载 `appassets.androidplatform.net`，但仍有 XSS 面 |
| `isLan` DNS 泄露 | 低 | `InetAddress.getByName(host)` 每次访问触发 DNS 查询，可能泄露连接意图 |
| 连接历史存完整 token | 低 | PrefsStore 历史记录包含 token，虽然加密存储，但增大攻击面 |
| token 只接受小写 hex | 低 | `[a-f0-9]{16,}` 拒绝大写 hex token，如果服务端生成大写 token 会解析失败 |
| 无 Content-Type 校验 | 低 | SSE 事件未校验 Content-Type，理论上可被注入非预期内容 |
| NetworkSecurityConfig 缺少 172.16/12 范围 | 中 | `ConnectionConfig.isLan` 通过 `InetAddress.isSiteLocalAddress` 识别 172.16.x.x 为 LAN，生成 `http://` URL，但 `network_security_config.xml` 未声明 172.16.0.0/12 允许明文，Android 默认阻止 → 该网段连接会失败 |

## 二、可维护性评估 — 68/100 ⚠️ 中等

### 核心问题

**1. 大文件过载** — 文件职责过重

| 文件 | 行数 | 问题 |
|------|------|------|
| `ClawdIcons.kt` | 664 | 手写 path data，应考虑代码生成或 SVG 资源 |
| `SettingsScreen.kt` | 647 | 应拆分为独立 section composable |
| `PetStateManager.kt` | 536 | 状态机 + 定时器 + 命令发射混杂，应拆分 |

| `SvgLoader.kt` | 450 | 大量硬编码 map，本质是配置数据，应外置为 JSON |

**2. 硬编码中文字符串** — 阻碍 i18n

至少 6 处 UI 文本直接写在代码中：
- `NavGraph.kt`: `"证书确认"`, `"信任并连接"`, `"取消"`
- `SettingsScreen.kt`: `"调整大小后需关闭再开启悬浮窗生效"`
- `SessionCard.kt`: `"等待中"` (dotColor/chipText 移动端覆盖)
- `ApprovalSheet.kt`: `translateSuggestionLabel()` 中的中文翻译

**3. 命名误导**

`WebSocketService` 实际使用 SSE 传输，类文档已说明但类名本身具有误导性。所有对它的引用 (`getWebSocket()`, `webSocketReady`) 都延续了这个错误命名。

**4. 死代码**

| 位置 | 代码 |
|------|------|
| `NotificationIcons.kt` | 整个文件零调用（`coloredCircleBitmap`, `coloredCircleDimBitmap`, `colorForState` 均无引用），建议删除整个文件 |
| `SvgLoader.kt` | `querySvgDimensions`, `hitTest` — public 但无调用方 |
| `ClawdIcons.kt` | `UserCircle` — 定义但未使用 |

**5. 颜色重复定义**

`Color.kt` 中 `ClawdBgDark == ClawdBackgroundDark`, `ClawdCardDark == ClawdSurfaceDark`, `ClawdCardBorderDark == ClawdBorderDark`，值完全相同但维护两份。此外 `ClawdCardDark(0xFF1A1D26) == ClawdSurfaceDark(0xFF1A1D26)`，同值不同名，分别用于 Card 容器和 Theme `surfaceColor`。

## 四、状态机评估 — 72/100 ⚠️

### PetState 状态机

16 个状态，优先级明确，PC 端对齐：

```
Error(8) > Notification(7) > Sweeping(6) > Attention(5) >
Conducting/Juggling/Carrying/Debugger(4) > Working(3) > Thinking(2) >
Idle-like(1) > Sleeping(0) > Sleep sequence: Yawning→Collapsing→Sleeping
```
> 注：相对顺序不变，报告原先的绝对值有误。

**优点**:
- `PetState` sealed class 设计清晰，`fromString` 容错（未知状态回退 Idle）
- Badge 消费机制（done 只触发一次 Attention，running 重置消费记录）
- `ONESHOT_STATES` 集合管理一次性状态自动回退
- `SLEEP_SEQUENCE` 集合包含 5 个状态（Yawning/Dozing/Collapsing/Sleeping/Waking），但实际发射序列仅 3 阶段（Yawning → Collapsing → Sleeping），Dozing 从未被显式发射，Waking 仅在中断睡眠时播放。60 秒空闲触发已确认。

**问题**:

| 问题 | 严重度 |
|------|--------|
| `PetStateManager` 过于复杂 (536行) | 高 — 状态解析、定时逻辑、命令发射三合一 |
| `resolveBestState()` 在 mutex 外读取 `sessionsFlow.value` | 中 — 理论上存在竞态 |
| Sleep 状态在优先级选择中被跳过 | 中 — 隐式行为，依赖注释而非类型系统保证 |
| 测试的是重建逻辑副本而非真实方法 | 高 — `PetStateManagerLogicTest` 测试的是复制的纯函数，不是实际 `PetStateManager.resolveDisplayState` |
| 双重超时机制 (SvgLoader 6s + PetStateManager autoReturn) | 低 — 文档化了但增加理解成本 |

### ConnectionState 状态机

5 个状态 (DISCONNECTED, CONNECTING, CONNECTED, RECONNECTING, AUTH_FAILED)，简洁明了。互斥不变量有测试保护。
